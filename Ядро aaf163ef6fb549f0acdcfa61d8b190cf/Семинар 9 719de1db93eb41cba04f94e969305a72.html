Семинар 9

  • файловые дескрипторы шарятся между родителем и ребенком, которые могут видеть изменения смещений
  • организация адресного пространства ребенка будет происходить в user space
  • подход copy on write копирует адресное пространство в процесс ребенка в readonly и копирует их только при записи

Вопросы 9 лаба:

  • 1. Рассказать об устройстве fork. Обратить внимание: Зачем нужна временная страница?

    когда мы делаем copy-on-write, мы выделяем новую страницу, мапим ее по временному адресу, копируем содержимое и мапим ее процессу уже по настоящему адресу, вместо той из которой копировали. Какое-то время она должна быть по другому адресу, чтобы мы могли адресовать одновременно обе страницы.

  • Зачем upcall пишется на ассемблере?

    Сначала работает C часть - _pgfault_handler, после которой необходимо вернуть регистры в изначальное состояние. Использовать jmp нельзя, тк он требует изменение значения регистров. Поэтому мы пушим старое значение rip на старый стек и выполняем ret, таким образом значение rip становится прежним

  • Как физически обеспечивается возврат разных кодов из sys_exofork() в родительском и дочернем процессе?

    sys_exofork поместит дочерний envid в регистр eax в соответствии с соглашением о вызовах для возвращаемого значения. Затем родитель сделает полную копию своей памяти дочернему элементу и сделает его runnable. После этого планировщик будет запускать потомок с помощью env_run () в какой-то момент. Когда дочерний процесс возобновляется, он попытается прочитать возвращаемое значение из eax, а eax сохраняется в структуре Env. Чтобы вернуть 0 ребенку, нам нужно изменить значение сохраненного регистра eax

  • 2. Каким образом организована обработка исключений в пользовательском пространстве? Зачем для этого нужен отдельный стек исключений? Когда он нужен?

    Fork в jos - юзерспейсный, и pagefault - должен обрабатываться с пользовательскими привелегиями. Если мы будем обрабатывать pagefault в пользовательском стеке, то при форке он будет copy-on-write, на copy-on-write стеке мы попадем в рекурсию.

    Стек исключений не переназначается таким образом. Вместо этого нужно выделить для него в дочернем процессе новую страницу. Поскольку обработчик ошибки страницы производит фактическое копирование страниц и при этом работает на стеке исключений, стек исключений не может быть сделан copy-on-write.
  • 3. Почему неэффективна стратегия COW?

    But the well-known downside to copy-on-write snapshot is that it will reduce performance on the original volume. That's because write requests to the original volume must wait to complete until the original data is "copied out".
